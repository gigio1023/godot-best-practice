## Principle

Godot scene files are source files, but they are serialized resources, not scripts. The agent should preserve their structure and use scripts or Godot APIs when direct editing becomes fragile.

Preferred control path:

1. Source data: resources, JSON, CSV, or project-specific definitions.
2. Behavior scripts: `.gd` or C# scripts that own runtime behavior.
3. Generator/editor scripts when scenes are large or repetitive.
4. Saved Godot resources/scenes.
5. Import, load, runtime, and visual evidence.

## Safer Direct Edits

Usually safe:

- `.gd` scripts
- C# scripts in existing C# projects
- JSON/CSV/layout data
- small `.tres` resources
- simple `.tscn` property edits after reading surrounding structure
- project-specific generator scripts

Higher risk:

- large hand-authored `.tscn` files
- `.scn` binary scenes
- imported `.glb` derived resources
- `.godot/` cache and imported artifacts
- animation tracks and complex subresources
- resources generated by addons

## `.tscn` And `.tres` Rules

Do not write GDScript syntax inside serialized resource files.

Avoid in `.tscn` and `.tres`:

- `var`
- `const`
- `func`
- `preload(...)` as script code
- arbitrary expressions

Use serialized resource forms already present in the file:

- `[gd_scene ...]`
- `[gd_resource ...]`
- `[ext_resource ...]`
- `[sub_resource ...]`
- `[node ...]`
- `ExtResource("id")`
- `SubResource("id")`
- property assignments

If unsure, read the official file format docs through `official-docs-index.md`.

## Direct Scene Edit Checklist

Before editing:

- inspect the file header and resource ids
- identify whether the scene is hand-authored, imported, or generated
- find attached scripts and dependent resources
- preserve resource ids unless deliberately updating all references
- avoid changing UID/path semantics without a follow-up Godot check

After editing:

- run import
- load the scene through Godot when possible
- verify required nodes and exported dependencies exist

## Generator Pattern

Use generator scripts when a scene is large, repetitive, data-driven, or likely to be regenerated.

EditorScript example:

```gdscript
@tool
extends EditorScript

func _run() -> void:
	var root := Node3D.new()
	root.name = "GeneratedWorld"

	var packed := PackedScene.new()
	var error := packed.pack(root)
	if error != OK:
		push_error("Failed to pack generated scene: %s" % error)
		return

	error = ResourceSaver.save(packed, "res://scenes/generated/generated_world.tscn")
	if error != OK:
		push_error("Failed to save generated scene: %s" % error)
```

For headless CLI scripts that do not need editor-only APIs, use a regular script run with `godot --headless --path . --script ...`.

## Ownership Marker

Generated output needs clear ownership:

```text
data/world_layout.json
tools/generate_world.gd
scenes/generated/world.tscn
```

If a generated scene was manually edited, move the intent back into the source data or generator before regenerating.

## Imported Assets

Do not edit `.godot/imported/` as source. Change:

- the source asset
- import settings
- importer script/addon
- post-import script
- material/resource overrides that are meant to be source-controlled

Check imported assets with `godot --headless --import`, then inspect scale, orientation, materials, animation clips, and collision setup.

## Resources As Data

Use resources for typed reusable data:

- `ItemDefinition.tres`
- `EnemyStats.tres`
- `DialogueGraph.tres`
- `ControlScheme.tres`

Use JSON/CSV when data needs external tooling, bulk editing, or language/runtime independence.

Remember resources are references. If a runtime instance mutates a shared resource, every user of that resource may see the mutation.
